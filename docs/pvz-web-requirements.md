# 植物大战僵尸网页版需求说明（Full 版，PC 优先）

## 1. 项目目标
- 打造一款在浏览器中运行的单人塔防游戏，玩法与经典植物大战僵尸核心体验相似。
- 首发平台为 PC 浏览器，保证主流现代浏览器可流畅运行。
- 资源采用占位素材与可替换规范，不直接使用受版权保护的原版素材。

## 2. 版本范围
- 版本目标：Full（较完整标准体验）。
- 包含：主菜单、选卡、对局、波次推进、结算、基础进度保存。
- 不包含：多人联机、无尽模式、编辑器、剧情章节系统。

## 3. 核心玩法循环
1. 玩家在开局阶段选择本局可用植物卡组。
2. 对局中通过掉落阳光与产阳植物累计资源。
3. 消耗阳光种植植物，阻挡并消灭持续来袭的僵尸。
4. 达成当前关卡全部波次后获胜；僵尸进入房子判定失败。

## 4. 对局系统需求
### 4.1 网格与地形
- 采用固定草坪网格（建议 5 行 x 9 列）。
- 每个格子最多放置 1 个植物（特殊植物可例外，作为扩展规则）。
- 僵尸按行推进，优先与本行植物发生交互。

### 4.2 阳光经济
- 阳光来源：自然掉落 + 向日葵生产。
- 阳光可被点击收集，计入玩家资源池。
- 植物种植前需校验：阳光余额、卡片冷却、格子可用性。

### 4.3 波次与出怪
- 每关由多个波次组成，包含普通波和大波提示。
- 每波按配置生成僵尸类型与数量，并设定行分布与出场时间。
- 关卡进度条展示整体波次推进。

### 4.4 胜负规则
- 胜利：清空最后一波且场上无存活僵尸。
- 失败：任一僵尸进入终点（房子）触发失败流程。

## 5. 实体系统需求
### 5.1 植物
- 基础类型至少包含：向日葵、豌豆射手、坚果墙、樱桃炸弹、寒冰射手、双发射手。
- 每个植物需定义：费用、冷却、生命值、攻击或效果、攻击间隔、作用范围。
- 植物状态至少包含：待机、攻击或触发、受击、死亡。

### 5.2 僵尸
- 基础类型至少包含：普通僵尸、路障僵尸、铁桶僵尸、撑杆僵尸、读报僵尸、铁门僵尸。
- 每个僵尸需定义：生命值、移动速度、特殊能力、受击反馈、死亡动画。
- 僵尸状态至少包含：行走、啃食、受击、死亡、特殊能力触发。

### 5.3 投射物与效果体
- 投射物至少支持：普通豌豆、冰豌豆（减速效果）。
- 需实现命中判定、伤害结算、穿透或销毁规则。
- 范围类技能（如樱桃炸弹）需支持区域伤害。

## 6. 交互与 UI 需求
### 6.1 主界面
- 包含开始游戏、继续游戏（有存档时）、设置入口。

### 6.2 卡槽与种植
- 显示卡片费用、冷却遮罩、可用或不可用状态。
- 支持点击卡片后在网格种植（或拖拽种植，二选一后固化）。
- 支持铲子模式移除已种植物。

### 6.3 对局信息
- 展示阳光总量、当前波次、进度条、暂停或倍速按钮。
- 大波来袭前显示明显提示。

### 6.4 结算页
- 胜利与失败页展示结果、耗时、击杀数、重开与返回菜单。

## 7. 数值与配置需求
- 所有植物、僵尸、波次数据配置化（JSON 或 TS 配置文件）。
- 平衡目标：新手可在前几关通过基础植物组合完成防守。
- 关键参数可热调（不改核心逻辑即可调数值）。

## 8. 音画与资源规范
- 使用占位素材，资源目录结构需规范（背景、角色、投射物、UI、音效）。
- 帧动画支持：按状态切换序列帧。
- 音频事件至少覆盖：种植、收集阳光、命中、爆炸、僵尸死亡、胜利或失败。
- 明确资源可替换接口，便于后续更换美术风格。

## 9. 非功能需求
- 性能：常规战斗场景目标 60 FPS（中端 PC 浏览器）。
- 兼容性：Chrome、Edge 最新稳定版优先支持。
- 稳定性：长时间运行（30 分钟）无明显内存泄漏与卡死。
- 可维护性：逻辑层与渲染层解耦，数据配置与代码分离。

## 10. 存档与设置
- 本地存档至少保存：解锁关卡、最近进度、基础设置（音量、倍速偏好）。
- 设置项至少包含：主音量、音效开关、暂停时行为。

## 11. 里程碑
### M1（MVP）
- 完成：基础网格、阳光、向日葵、豌豆射手、普通或路障僵尸、基本胜负。

### M2（Beta）
- 完成：选卡、冷却、更多植物僵尸、波次进度、结算页、基础音效。

### M3（Full）
- 完成：完整目标清单、性能优化、存档设置、视觉与交互打磨。

## 12. 验收标准
- 功能验收：核心循环可完整进行且无阻断。
- 规则验收：阳光、冷却、碰撞、波次、胜负判定符合配置预期。
- 体验验收：关键交互反馈清晰，普通用户可在无说明下完成首关。
- 性能验收：目标环境下帧率与稳定性达到非功能需求。

## 13. 风险与缓解
- 风险：同屏对象过多导致掉帧。
- 缓解：对象池、渲染裁剪、降低非关键特效更新频率。
- 风险：状态机复杂造成逻辑错误。
- 缓解：统一状态机定义与事件流，补充自动化规则测试。

## 14. 版权与合规声明
- 本项目需求仅描述玩法机制，不包含任何原版受版权保护素材。
- 所有图像、音频、字体需使用自制、可商用授权或开源许可资源。

## 15. 执行拆分（可直接交接开发）
### Phase A：框架与底座（预计 2-3 天）
1. 初始化前端工程与构建链路，建立游戏主循环、场景管理、资源加载器。
2. 建立网格坐标系统、实体注册系统、时间与事件总线。
3. 打通最小可渲染链路：草坪背景、僵尸移动、植物静态摆放。

验收要点：
- 可进入对局场景并稳定运行 5 分钟无异常。
- 资源加载失败时有降级提示，不阻塞页面。

### Phase B：核心战斗闭环（预计 4-6 天，依赖 Phase A）
1. 实现阳光掉落、收集、资源扣费逻辑。
2. 实现卡槽冷却、植物种植校验、铲除逻辑。
3. 实现豌豆射手与普通僵尸交互（命中、掉血、死亡）。
4. 实现失败判定（僵尸入屋）与胜利判定（波次清空）。

验收要点：
- 可完整进行一局基础关卡并结束。
- 阳光、冷却、碰撞、伤害与判定规则符合配置。

### Phase C：内容扩展与表现（预计 5-8 天，依赖 Phase B）
1. 扩展植物：向日葵、坚果、寒冰射手、樱桃炸弹、双发射手。
2. 扩展僵尸：路障、铁桶、撑杆、读报、铁门。
3. 增加大波提示、进度条、结算界面、暂停与倍速。
4. 增加关键音效与状态动画（攻击、受击、死亡、爆炸）。

验收要点：
- 至少 3 关可玩，难度递进可感知。
- 大波提示、结算页、暂停恢复流程无阻断。

### Phase D：Full 收尾与质量（预计 3-5 天，可与 Phase C 后期并行）
1. 性能优化：对象池、更新裁剪、渲染批处理。
2. 存档与设置：本地保存关卡进度、音量、基础偏好。
3. 回归测试与平衡调优：首关通过率、中关卡压力、异常输入处理。

验收要点：
- 目标设备稳定接近 60 FPS。
- 连续游玩 30 分钟无明显内存增长异常。

## 16. 测试用例清单（最小必测）
1. 阳光不足时种植失败且 UI 明确反馈。
2. 冷却未结束时卡片不可用，结束后自动恢复。
3. 同格重复种植时拒绝并提示。
4. 冰豌豆减速可叠代或刷新策略符合设计定义。
5. 樱桃炸弹范围边界命中准确，无跨行异常。
6. 僵尸啃食植物至死亡后恢复前进状态。
7. 最后一波清空后胜利，不被延迟生成单位打断。
8. 失败触发后停止出怪与输入，允许重开。
9. 刷新页面后存档可恢复至最近关卡。
10. 在高压场景（多僵尸 + 多子弹）下无明显卡顿尖峰。

## 17. 交付物定义
- 需求文档（本文件）。
- 数值配置表（植物、僵尸、波次）。
- 资源占位清单与替换规范。
- 测试用例与验收记录模板。

## 18. 范围冻结建议
- 冻结点 1（MVP 完成后）：不新增单位类型，只修规则问题。
- 冻结点 2（Beta 完成后）：不新增系统能力，只做体验打磨与性能优化。
- 冻结点 3（Full 提测前）：仅允许阻断级缺陷修复。
